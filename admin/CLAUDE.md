# CLAUDE.md — Modiq Admin 개발 가이드

> 이 문서는 Claude Code가 **Modiq 어드민(플랫폼 운영자 콘솔)** 을 개발할 때 항상 먼저 읽고 따라야 하는 기준 문서다.
> 저장소 루트 `CLAUDE.md`의 공통 규칙(브랜치/머지 흐름·보호 영역 등)과 **함께** 적용하며, 충돌 시 **보안 규칙이 우선**한다.
> 목표는 단 하나: **오류 없는 어드민 개발.** 모든 기능은 아래 5단계 게이트를 통과해야 한다.

---

## 0. 프로젝트 개요

- **제품**: Modiq — 모델 에이전시 관리 B2B SaaS.
  이 어드민은 **Modiq SaaS 자체를 운영하는 플랫폼 운영자(슈퍼관리자) 전용 콘솔**이며, 고객(에이전시)이 쓰는 앱과 **코드·배포가 완전히 분리**된다.
- **운영**: 1인 창업 (Jae / 주식회사 스페이스재형)
- **스택**: 정적 어드민 프론트(빌드리스 단일 HTML, 필요 시 별도 Vite) + Supabase **Edge Function `admin-api`**(service_role 보유, 서버 측) + Postgres(RPC + RLS)
- **저장소**: `aimoagency/modiq` — 어드민 코드는 **`admin/`** 디렉터리
- **배포**:
  - 고객 앱 = Vercel(웹) + Netlify(모바일), 운영 브랜치 `master`
  - **어드민 = 별도 정적 배포(예: `admin.modiq…` 서브도메인)** — 고객 앱 번들과 절대 섞지 않는다.
- **버전 기준**: 고객 앱 **v1.1.0** / 어드민 **v0.1.0 (신규)**
- **개발 환경**: Claude Code (web) + GitHub
- **Supabase 프로젝트**: `fijtpyrmqzjefucsqfos`

### 어드민이 다루는 대상 (현재 범위)
- **테넌트(에이전시) 관리**: 목록 · 대표 이메일 · 플랜(trial/starter/standard/pro) · trial 만료 · 가입일
- **기능별 사용량**: 멤버 · 모델 · 섭외 · 고객사 · 패키지 수
- **저장용량(하드)**: `model-photos` 버킷의 **에이전시별 파일 수·용량**(상위 폴더 = `agency_id`)
- **계정 정리**: 고아 auth 계정(에이전시·멤버십 없음) 조회·삭제
- **플랜 변경**: `set_plan`(허용값만)
- **(예정)** 결제·요금제 연동, 운영 지표(가입 추이·활성·MRR), 공지·기능 플래그

### 백엔드 구성 (이미 적용/설계됨)
- 테이블 **`platform_admins(email unique)`** — **RLS ON · 정책 없음 → service_role만 접근.**
  비어 있으면 아무도 어드민 권한 없음(안전). 운영자 이메일을 등록해야 활성화된다.
- 함수(**SECURITY DEFINER**, `anon`/`authenticated` 실행 권한 revoke):
  - `admin_tenant_stats()` — 에이전시별 플랜·기능 사용량·저장용량
  - `admin_orphan_accounts()` — 고아 계정 목록
- Edge Function **`admin-api`**(`verify_jwt=false`, **자체 인증**):
  호출자 JWT로 `auth.getUser()` → `platform_admins` 확인 → service_role로 처리.
  액션: `tenants` / `orphans` / `delete_user`(고아만) / `set_plan`(plan 화이트리스트).

## 1. 절대 규칙 (위반 시 즉시 중단)

1. **설명용 텍스트를 코드 파일에 섞지 말 것.** 박스문자·표·자연어 설명이 `.ts`/`.tsx`/HTML 스크립트 본문에 들어가면 빌드/실행이 깨진다. 설명은 주석(`//`, `/* */`) 또는 별도 `.md`로 분리.
2. **`service_role` key를 어드민 프론트에 절대 두지 말 것.** 프론트엔드에는 로그인한 운영자의 JWT와 공개 `anon key`만. 권한이 필요한 모든 작업은 **`admin-api` Edge Function 경유**.
3. **모든 권한 작업은 `platform_admins` 화이트리스트 검증을 통과해야 한다.** RLS를 우회하는 직접 쿼리 금지. 전(全) 테넌트 조회·변경은 **SECURITY DEFINER 함수 + admin-api로만**.
4. **파괴적 동작에는 안전장치 필수.** 계정 삭제는 **'고아(에이전시·멤버십 없음)'만** 허용, 플랜 변경은 **허용값만**. 삭제·일괄 작업 전 확인 절차.
5. **타입 단언(`as any`, `@ts-ignore`)으로 에러를 덮지 말 것.** 근본 원인 해결.
6. **검증되지 않은 변경을 운영(`master`)에 머지하지 말 것.** 5단계 게이트 통과 후 진행.
7. 컴포넌트·함수·테이블·컬럼명을 **추측으로 지어내지 말 것.** 확실하지 않으면 먼저 코드베이스/DB를 검색해 실제 이름을 확인한다.

## 2. 개발 5단계 게이트 (모든 기능은 이 순서로)

이전 단계가 끝나야 다음 단계를 시작한다. 건너뛰기 금지.

### ① 서비스 기획 (PLAN)
- 이 기능이 **운영자의 어떤 문제**를 푸는지 한 문장으로.
- 입력 → 처리 → 출력 흐름을 글로 적는다(코드 X). 어디까지가 프론트, 어디부터 `admin-api`인지 경계를 명시.
- 영향받는 테이블·함수·액션을 나열.
- **엣지 케이스**: 테넌트 0건 · `platform_admins` 비어있음(전원 403) · 고아 0건 · 비(非)관리자 접근 · 저장용량 null · 대량 테넌트 · 네트워크 실패 · `delete_user` 대상이 고아 아님.

### ② 사용자 편의 (UX)
- 운영자(=Jae) 관점에서 **클릭 수↓·실수 방지** 설계.
- 필수: 로딩/빈/에러 상태 UI, **위험 동작(계정 삭제·플랜 변경) 확인 모달**, 결과 0건 안내, 모바일 터치 타깃 44px↑.

### ③ 코딩 개발 (BUILD)
- 기획·UX 범위 **안에서만** 구현. 임의 확장 금지.
- 새 액션은 admin-api의 인증·화이트리스트 검증 흐름을 그대로 따른다.
- 새 RPC/마이그레이션은 RLS·권한(revoke)과 함께 설계, `supabase/`에 기록.
- 작은 단위로 커밋.

### ④ 검수 (REVIEW)
- **빌드/타입 0 에러.**
- 체크리스트: 절대 규칙(섹션 1) 위반 없음 · 모든 Supabase/Edge 응답의 `error` 처리 · 하드코딩 키 없음(허용: 공개 URL·anon key) · **service_role 미노출 재확인** · 죽은 코드/미사용 import 제거.

### ⑤ 사용성 테스트 (TEST)
- 정상 경로 1회 + 엣지 케이스 전부 실제 검증.
- 최소 시나리오: **비관리자 접근 403** · 고아만 삭제(비고아 거부) · 플랜 허용값만 · 통계/저장용량 정확성(샘플 대조) · 대량 테넌트 렌더 · 외부 실패 graceful.
- **회귀**: 이번 변경이 **고객 앱(고객 데이터·정산·달력·알림)** 에 영향 없는지 확인. 어드민은 고객 앱과 분리이므로 코드 침범 금지.

> ⑤까지 통과해야 "완료". 하나라도 실패하면 그 단계로 복귀.

## 4. 코드 컨벤션

- **언어**: TypeScript strict(Edge Function·별도 Vite 시). `any` 금지. 단일 HTML 프론트도 로직은 함수 단위로 분리.
- **상태**: 로컬은 `useState`/모듈 변수. 서버 상태는 admin-api 호출 레이어로 분리.
- **네이밍**: 컴포넌트 PascalCase, 함수·변수 camelCase, 상수 UPPER_SNAKE_CASE. **액션명·플랜값은 상수화**.
- **Supabase/Edge 호출**: 항상 `const { data, error } = await ...`(또는 fetch 응답 `ok`/`error`) 패턴, `error` 즉시 처리.
- **주석**: 설명은 코드 위 주석으로. 코드 본문에 자연어/표/박스문자 금지.
- **매직넘버·문자열**: 상수/설정으로 추출.
- **날짜/타임존**: 한국 기준(Asia/Seoul) 명시. trial 만료 등 경계값 주의.

## 5. Supabase / 보안 규칙

- **권한 부여는 `platform_admins` 화이트리스트로만.** 신규 운영자 추가 = 이 테이블 insert(service_role/운영자 SQL). 2FA 권장.
- `service_role`은 **`admin-api`(서버)** 에만. 프론트엔드는 anon key + 운영자 JWT만.
- 전 테넌트 접근용 SECURITY DEFINER 함수는 **`anon`/`authenticated` 실행 권한 revoke** 유지, `search_path=''`+스키마 정규화로 작성.
- 모든 신규 테이블 **RLS 활성화**. 어드민 전용 테이블은 정책 없이 두어 service_role만 접근(또는 운영자 한정 정책).
- 고객 **민감정보(주민번호·계좌·사업자번호)** 는 어드민에서도 **마스킹·최소 노출·접근 로깅**(`secure_id_access_log`) 원칙. 평문 복호화는 기존 RPC·감사 경로로만.
- 마이그레이션은 reversible하게 작성하고 **적용 전 Jae 확인**.

## 6. 커밋 / 배포 규칙

- 개발은 작업 브랜치 `claude/*`에서. **5단계 게이트 + 빌드/타입 통과** 후, 루트 `CLAUDE.md`의 머지 흐름(`claude/* → feature/package → master`, **merge commit**)을 따른다.
- 커밋 메시지: `타입: 요약` (예: `feat: 어드민 테넌트 사용량·저장용량 표시`, `fix: 고아 계정 삭제 가드 보강`). 타입: `feat / fix / refactor / style / docs / chore`.
- Edge Function·마이그레이션 변경은 Supabase(MCP)에 배포 후 **`supabase/`에 SQL·함수 본문을 보관**(버전관리).
- 배포 후 **운영자 계정으로 어드민 핵심 경로 1회 스모크 테스트**(로그인 → 테넌트 조회 → 사용량/용량 표시 → 비관리자 403 확인).
